昨天我写到“会聊天”和“能行动”之间隔着一条工程轨道。今天继续往里挖，发现真正难的不是再加一个工具，也不是再接一个更强的模型，而是给整个助手设计一颗稳定的心脏。今天这颗心脏在代码里叫 Reactor。

刚开始做 ai-node 时，`AgentCore.chat()` 是一个很好理解的入口：收到用户输入，拼 system prompt，带上当前待办和偏好，请求 LLM，模型如果返回 tool_calls，就校验参数、执行工具、再把工具结果送回模型。这个流程直观，也适合第一个版本。但当项目开始有 SQLite、短期记忆、长期偏好、Codex 接入、校验重试和流式输出以后，一个函数继续扛所有事情就会变得危险。危险不在于它不能运行，而在于它越来越难解释。

所以今天最重要的变化，是把推理循环拆成了三个概念：事件、状态和副作用。

事件是外部世界发生了什么，比如用户说了一句话、LLM 返回了一条消息、工具执行完成。状态是当前这一轮对话进行到哪里了，比如消息列表、校验失败次数、工具调用轮次、最终回复和停止原因。副作用则是下一步要做什么，比如请求 LLM 或执行工具。

这个拆法最打动我的地方，是它把“决定”和“执行”分开了。`AgentReactor.step(state, event)` 只负责判断：现在该请求模型，还是该执行工具，还是该因为参数错误让模型重试。它不直接访问网络，不写数据库，也不操作终端。真正的网络请求、工具执行和错误恢复交给 `AgentRuntime` 处理。工程上可靠的自主，不是让模型在黑盒里自由发挥，而是让每一次动作都落在明确的状态转换上。

今天的测试也很有意思。`tests/test_reactor_runtime.py` 不是去模拟一个完整聊天，而是逐个验证状态机行为：用户消息会产生 `RequestLlm` effect；普通助手回复会进入 final 状态；合法 tool call 会变成 `ExecuteTools`；非法参数不会执行本地工具，而是追加 `role=tool` 的错误反馈，再请求模型重新生成。工具参数校验失败时，工具尚未执行；错误信息通过 Chat Completions 能理解的 `role=tool` 协议回给模型；超过重试次数就停住，避免错误参数写入本地待办。这不是提示词技巧，而是安全边界。

短期记忆也在这个结构里变得更清楚。`SessionMemory` 只保存 user 和 assistant 的自然语言消息，不保存 tool_call 和工具结果。今天再看这个选择，我觉得它比一开始想象得更重要。个人助手需要记住“我刚才说了什么”，但不应该把每个数据库返回、每个校验错误都塞进对话窗口。真实待办和长期偏好应该从仓储读取，工具链噪音应该留在执行链里。

Codex 接入这一段也暴露了本地助手的复杂度。README 里现在写得很明确：默认 LLM 后端是 OpenAI 兼容 API Key，Codex 登录态只是可选后备；Codex 模式优先使用 `app-server --listen stdio://`，并把 auth 复制到项目私有的 `data/codex_home`，只复用登录态，不加载用户个人配置、插件、MCP 和 hooks。这个设计不是为了炫技，而是为了隔离边界。

今天还有一个现实取舍：带工具时仍然保守地关闭流式工具链。流式输出当然更像真人聊天，但当前工具循环依赖完整的 `message.tool_calls`。如果为了“看起来更快”而破坏工具调用闭环，最后得到的是半截回答和不可验证的操作。

把第 3 天的进展串起来，我看到 ai-node 正在从“一个能调用工具的 CLI demo”变成“一个有主循环的个人助手雏形”。SQLite 负责事实，SessionMemory 负责短期上下文，ToolExecutor 负责本地动作，LLM client 负责模型适配，而 Reactor 负责把它们编排成一条可测试的行动链。

这也改变了我对“成熟”的定义。成熟不是代码变多，也不是模型名字变新，而是每个边界都能被解释：为什么这里要校验，为什么这里不能执行，为什么这里要重试，为什么这里要停止。个人 AI 助手的主循环必须先于炫酷能力稳定下来。没有主循环，记忆、工具、RAG、MCP 都只是散落的零件；有了主循环，它们才有机会被接进同一套状态机里，成为可观察、可测试、可演进的能力。

所以第 3 天的标题我想叫“Reactor 让助手开始有主循环”。从今天开始，这个项目不只是让模型回答我，而是在给模型搭一条能长期运转的执行神经。
