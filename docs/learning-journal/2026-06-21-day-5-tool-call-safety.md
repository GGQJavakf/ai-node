第 5 天，我把注意力从“助手知道哪些工作”拉回到一个更底层的问题：它真的能可靠地执行工具吗？

前几天我一直在补个人 AI 助手的记忆、工作项和 Codex 日报入口。这些能力让助手越来越像一个本地工作台。但只要它还会替我新增待办、更新任务、读取工作状态，工具调用这条链就不能只是“模型说了算”。今天的仓库变化提醒我，一个 Agent 最危险的地方往往不在回答错一句话，而在拿着错误参数去改本地数据。

我之前对工具调用的理解比较直接：模型返回 tool call，本地执行器按名字找到函数，执行完把结果发回模型，再让模型组织自然语言回复。这个流程看起来像标准 Function Calling 教程，但真正做成个人助手后，细节会马上变复杂。模型可能把优先级写成不支持的值，可能漏掉标题，可能把空字符串当成时间，也可能在需要工具调用时走了流式文本路径。每一个小偏差，都会在本地数据里留下脏记录。

所以今天的重点不是增加新工具，而是把工具调用变成一条有护栏的执行链。

第一层护栏是参数模型。工具定义不能只给模型看，还要被本地执行前校验复用。这样做的好处很明显：模型可以自由生成，但本地系统不必相信它。新增待办时，标题必须存在，优先级只能落在固定枚举里，可选时间如果只是空白字符串，要按未声明处理，而不是写入一个看似合法、实际没意义的值。这个边界一旦立住，Agent 的风险就从“模型可能直接污染数据”降到“模型需要重新生成合法参数”。

第二层护栏是失败回灌。以前我容易把校验失败理解成一次普通异常，报错就结束。但 Chat Completions 的工具协议有一个很关键的事实：如果模型发起了 tool call，后续消息里必须给这个调用补上对应的 tool 输出。哪怕这次没有真正执行工具，也要把“参数错误”作为 tool 消息回填，并带上原来的调用 ID。否则下一轮请求就可能断在协议层，出现类似“找不到 tool output”的问题。

这件事让我对 Agent 循环的理解更具体了：工具调用失败不只是业务失败，它还是对话协议状态的一部分。协议状态没补齐，模型就没有办法稳定地重新生成。今天的代码演进把这块拆得更清楚：参数不合法时，不执行本地工具；把错误作为 tool 输出写回消息；再追加一条明确要求重新生成合法 tool call 的用户消息。这样模型有纠错上下文，本地数据也没有被动过。

第三层护栏是 Reactor 主循环。把用户消息、LLM 响应、工具执行结果都当成事件，再由一个明确的状态机决定下一步 effect，这比在一个大函数里写满 if/else 更容易看清责任边界。用户输入触发一次 LLM 请求；LLM 返回普通文本就结束；LLM 返回工具调用就先校验；校验通过才执行工具；工具结果再进入下一轮 LLM。失败次数、工具轮数、最终停止原因都挂在状态上，不再散落在临时变量里。

我今天最有感触的一点是，Reactor 并不是为了显得架构高级。它解决的是“我怎样证明这条链路不会悄悄越界”。当工具参数错了，状态机会停在校验失败分支；当工具轮数太多，运行时会按上限终止；当网络或 HTTP 请求失败，最终响应里会带上明确错误。对于个人助手来说，这种可解释的停止，比盲目继续执行更重要。

短期记忆也在这里变得更实用。仓库现在保留一个滑动窗口，只记录用户和助手的自然语言消息，不把完整工具调用和工具结果都塞进去。这个取舍很克制。真实待办状态应该来自本地仓储，工作事实应该来自同步快照，短期记忆只负责让“刚才那个”“继续”这类表达有上下文。把记忆做得太重，反而会把工具结果、错误栈和噪声一起塞进模型窗口，后面更难判断模型到底依据什么在行动。

LLM 接入也在继续变稳。默认路径仍然是 OpenAI 兼容接口，但 Codex 登录态也被纳入可选后备：优先尝试 app-server，必要时回退到 exec，并且把 Codex home 隔离到项目本地，只复用登录身份，不加载个人全局配置、插件、规则和 hooks。这一点对我很重要。个人助手调用模型时，应该尽量减少环境噪声，否则同一段代码在不同机器、不同全局配置下可能表现完全不同。

今天我还看到 HTTP 重试被补上了。429、500、502、503、504 这类可重试错误和 401 这类不可重试错误不能混在一起处理。前者可以退避重试，后者应该尽快暴露配置问题。这个细节不大，但它代表一个思路：AI Agent 不是只和模型打交道，它也在和网络、命令行、本地文件、登录态、测试替身打交道。每个外部边界都要分清“暂时失败”和“确定失败”。

测试给今天的学习收了口。新增的测试没有追求大而全，而是盯住几个容易出事故的点：参数校验失败会重试；失败的 tool call 会收到对应 tool 输出；未声明开始时间会默认到当前时间；带工具的流式请求会退回非流式；清空历史会清掉短期记忆；Codex 客户端会使用隔离参数；HTTP 可重试错误会重试，不可重试错误不会乱重试。这些测试像一组路标，告诉我 Agent 的可靠性不是一句“我们做了校验”，而是每个边界都有可回归的证据。

第 5 天的结论很朴素：个人 AI 助手越接近真实执行，越不能把智能全部交给模型。模型负责理解意图和生成候选动作，本地系统负责校验、执行、记录和停止。工具调用不是魔法，它是一条协议、一组参数模型、一个状态机、一套错误恢复策略，再加上能反复跑的测试。

我现在更愿意把 ai-node 看成一个逐渐长出骨架的助手。第 1 天它开始有记忆，第 2 天它开始能行动，第 3 天它有了主循环，第 4 天它能看见工作现场，第 5 天它开始学会在行动前刹车。这个“刹车”并不妨碍自动化，反而是自动化能长期跑下去的前提。

下一步，我会继续沿着这个方向看：当工具越来越多，哪些能力应该只读，哪些可以写入，哪些必须要求证据；当工作项、偏好、日报和待办都进入同一个助手，如何让它既能主动建议，又不会越权替我完成外部系统操作。今天先把工具调用别掉链这件事做扎实。
